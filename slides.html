What is version control anyway?

Defintion of Version Control

Ver·sion Con·trol:

The task of keeping a software system consisting of many versions and configurations well organized.

There're a number of different applications that try to solve this problem:

git
mecurial
subversion

Before we examine these individually, we're going to take a brief detour and look at what a version control system actually is.

What does Version Control software do?

Take a step back and ask yourself: How would you create as a program to manage versions of your data?

Ignore everything you've heard about SCM (Source Control Management), just think about how you would do it.

  • What backup features does it have?

  • How does it store your data?

  • Where does it store your data?

    • Where does it store your data on your pc?

    • Does it store your data on a remote server? If so, how?

  • What's more important: ease of use or efficiency?

Git, and other VC systems, each answer these questions in their slightly different ways.

A possible Chain of Thought

You may have come up with creative, perhaps even algebraically sound systems between the slides, but for our purposes we will guide you along a simple path that seems to be a plausible evolution of a VCS:

  • Folder Backup
  • Indexed Folder Backup
  • Hashed Folder Backup

The naive option: Folder Backup

If you'd never heard of Version Control and you wanted to make sure your work was backed up, you'd probably make a backup folder somewhere on your drive and simply copy your files into it

This is a form of version control: you are managing two versions of your data, one in your working folder and the other a snapshot in a backup folder

If we made a reasonably sophisticated programme to do this, it would probably be able to:

  • Back your data up into a folder (like .backup)

  • Restore your data from a folder over your current one

      • realise you made a mistake some time ago and want to revert?

    Multiple backup folders

    The version control approach that most sufficiently cautious people tend to replicate: copy multiple versions of your data into different, numbered folders.

    This approach greatly extends the capabilities of the previous version control software as instead of having only one version to examine to see how your data is changing, you can now have as many as you want

    If we made a programme to do this, it should:

    • Allow you to make multiple copies of your code in a folder

    • Restore your code from one of your backups over your current one

    The basic option: Backup

      def backup(self):
        """
        Backup into .backup/contents/x where x is the number of repos plus one.
        """
        if not vc.is_init(self.backup_path, [self.contents_path],
                self.index_path, self.identity_string):
            print(self.not_exist_msg)

        backup_no = str(len(self.get_backups())+1)
        os.mkdir(os.path.join(self.contents_path,backup_no))
        files = os.listdir()
        files.remove(self.backup_path)
        vc.cp(zip(files,files),".",os.path.join(self.contents_path,backup_no))
        with open(self.index_path,"a") as f:
            f.write("\n%s;%s"%(backup_no,str(datetime.now())))
        return True

    Problems with just using multiple folders

    While this is definitely a major improvement over our first version, there are still a number of outstanding issues

    • It wastes a collosal amount of space
      • Simply renaming a file ensures that it is copied to the backup folder when clearly nothing about its content has changed
      • By the very nature of file-oriented work, most data is unchanging between backups
      • This means that by backing up it's very possible that you are creating duplicates of certain files (completed code &c.)
    • How would you track changes between iterations or know when their contents have changed?
    • If you wanted to colloborate with this programme, how would the fact that the versions are only numbered relative to your backups impact collaboration?

        • It would seem that these problems all stem from a lack of an underlying structure. Enter the hashing scheme.

    The saviour of our version control endeavour: hashing

    Otherwise known as the approach that seems obvious in retrospect.

    In order to eliminate file duplication and provide assurance for file integrity, we hash every file before it's backed up.

    This also allows very easy sharing: if we wanted to sync our copy with a centralised repository, we can simply compare our hashed file listings per backup and see where they diverge.

    Note: this is the first version iteration that you can't do manually. In order to have the most efficient solution, we have to rely on a program doing this for us.

    Hashing VCS: Backup and Restoration

    We would like to implement a programme to:

    • Verify the validity of the backups:

      • Hash every file in the backup directory and make sure its hash matches its filename.

      • Make sure every hash listed in the catalogue is present in the backup directory.

    • Allow you to backup multiple versions of your code using the following process:

      • Hash all the files in your directory.

      • Create a catalogue which pairs filenames with their hashes - this forms the record of the backup.

      • Backup all files whose hashes aren't already present in the backup folder. To make this easier, when backing up make the filename the hash.

    • Restore your code from one of your backups by:

      • Clear the working directory

      • Loading the catalogue of the relevant backup

      • Copy the files whose hashes are listed in the catalogue into working directory

    Hashing VCS Implementation: Part I

        def backup(self):
        # ... check that the repo exists ... 
        backup_no = str(len(self.get_backups())+1)
        # Hash things and checking if we have them
        existing = set(os.listdir(self.contents_path))
        current_files = []
        directory_structure = []
        for dirname,subdirs,fnames in os.walk("."):
            # Let's not recurse into .hfk
            if self.backup_path in subdirs:
                subdirs.remove(self.backup_path)
            #grab hash of files
            for f in fnames:
                current_files.append(os.path.join(dirname,f))
            # store the directory structure
            for d in subdirs:
                directory_structure.append(os.path.join(dirname,d))
        pairs = [(f,self.careful_hash(f)) for f in current_files]
        # Let's create the catalogue file first, a simple dump of pairs
        catalogue = open(os.path.join(self.revisions_path,backup_no),"w")
        catalogue.write('\n'.join([p[0]+' '+p[1] for p in pairs]))
        catalogue.close()
        # and we need to dump the dir structure
        structure = open(os.path.join(self.revisions_path,backup_no+"d"),"w")
        structure.write('\n'.join(directory_structure))
        structure.close()
        # Copy accross all the files listed in pairs but not currently in files/
        files = []
        for fle,hsh in pairs:
            if not hsh in existing:
                files.append((fle,hsh))
                print("--> "+fle+" is new or has changed, storing")
        if files == []:
            print("No changes, bailing.")
            return True
        vc.cp(files,".",self.contents_path)
        
        with open(self.index_path,"a") as f:
            f.write("\n%s;%s"%(backup_no,str(datetime.now())))
        return True

    Hashing VCS Implementation: Part II

         def restore(self,revision=-1):
        # ... validate repo ...
          
        # If -1, find the last revision
        if revision==-1:
            revision==len(open(self.index_path,"r").read().split('\n'))

        # pull up the catalogue of files and directory structure files
        catalogue = os.path.join(self.revisions_path,str(revision))
        structure = os.path.join(self.revisions_path,str(revision)+"d")
        if not (os.path.exists(catalogue) and os.path.exists(structure)):
            print(self.no_existing_backup)
            return False

        # remove all files in the current dir
        del_files = os.listdir(".")
        del_files.remove(self.backup_path)
        vc.rm(del_files, ".")

        # rebuild structure from the file
        dirfile = open(structure,'r')
        dirs = dirfile.read().split('\n')
        dirfile.close()
        for d in dirs:
            os.mkdir(d)

        # generate pairs of files and hashes (actually file names in files/)
        catfile = open(catalogue,'r')
        files = [(p[1],p[0]) for p in [line.split(' ') for line in catfile.read().split('\n')]]
        catfile.close()
        
        # simply copy them to the current directory
        vc.cp(files,self.contents_path,".")
        return True

    Where to from here?

    Where we go from here depends on how we want to answer the question depends entirely on how we answer the question "where is the master repository located?". That is, if it resides only on some centralised server then we have invented CVS (centralised version control system), but if everyone is free to keep a copy anywhere, then we have created a DVCS (distributed version control system).

    Centralised VC's

    CVS, Subversion (as used by Amazon) and co.

    • Centralised version control systems all have, as their defining characteristic, the master copy containing all revision information is located only on the master server on only one machine. Thus, all developers must commit to and pull from it. This introduces a distinct heirarchy of repositories: the central repository is official one, any code that is on your machine or not in that repository isn't official.

    • The implication of such a system is that anyone who has the data does not actually have the repository, they merely have a copy of the then current version. Thus, in order to acquire past versions and effectively track changes and information one need necessarily be online and connected to the server

    • Finally, most centralised services make it difficult to do tasks like branching (covered later).

    Distributed VC's

    We'll be focusing on git here, but note that there are a number of different distributed VCs, such as Darcs and Mecurial.

    • Decentralised, this means that while there might be centralised repositories, when we get the data from that repository we don't get a version, we get the whole repository. This is essentially the model we've developed.

    • Thus there is no technically privledged repository: anyone who gets a copy of a repo has an identical copy to everyone else

    • The way git does this is by storing the entire repository, with all its backups, hashed in a .git folder stored in the working directory, very much like our hashed VCS

    Internal structure of git

    In fact, git has a lot in common with our little toy hashed vcs structure-wise. This isn't a course on the interals of git, but to give a brief overview:

    • It stores all its objects (including files) in one folder with their filenames as their SHA1 hashes
    • Every time you make a backup, it generates a series of trees, each representing a folder being backed up by listing the pairs of the contents of each file along with their hash (like our catalogues structure)
    • A backup is stored as a commit file which points to the tree representing the working directory at the time of backup (exactly like our catelogue files)
    • Commits are then linked together to provide a history of the repository.
    Although we will not be concerning ourselves with anything finer than the structure of commits from this point onward, it is worth noting that in three simple iterations we came up with something that stores data similar to one of the most popular SCM systems at the moment, an SCM that some call complex!

    So, what have we achieved?

  • We've developed a toy VCS from scratch teaching us a couple of important lessons:

    • Doing VCS manually is as silly as it's inefficient and very much lacking in essential features compared to an actual SCM system

    • Every VCS is simply an approach to provide a common feature set, they just go about it in a number of different ways.

    • If we didn't have to waste so much time doing CS assignments, we probably would have come up with git by ourselves!

  • We've learnt about the differnce between centralised and distributed version control

    Staging: tracked and untracked

    Git has two notions of file monitoring: tracked and untracked. Untracked files are precisely those about which git does not care and which will not be incorporated into any commits or revisions. Tracked files are those that git concerns itself with. To tell git to track a file, we use git add .

    The file is now said to be staged for a commit, that is, git has noted that there are some changes in the file which are not reflected in the current state of the repo up to the last commit and is prepared to record these changes

    When staging a file, In particular, git hashes it, compares its hash to the manifest and decides what, if anything, is new about this file

    There are a number of reasons why you'd like this level of granularity, e.g. not commiting large binary files that are automatically created when compiling or wanting to have some testing data you don't want added to the whole folder.

    In order to find out more about the files in your repository, type in git status. It'll give you the files and directories aren't yet tracked and a reminder of how to get them to be tracked.

     
    $ git status
# On branch master
#
# Untracked files:
#   test.file

    To stage a file, we use git add

     
    $ git status
# On branch master
#
# Untracked files:
#   test.file
$ git add test.file
$ git status
# On branch master
# Changes to be committed:
#   new file:   test.file

    Commit: how to make snapshots

    Once all the relevant files have been staged what remains is to tell git that we are ready to bundle up all the changes we have made into one package - a commit.

    This can be likened to taking a discrete snapshot of the state of the repo and storing it, so that at any point in the future we may go back and examine how things were, or branch from a particular state of the repository and so on. This is the core feature of any VCS.

    When taking such a snapshot, we must provide git with a commit message. This message outlines the reasons for capturing this particular stage, and any relevant changes that have taken place since the last one.

    To do this, we issue git with the 'git commit' command. If there are no staged files, git will not make a commit. If there are, then git notes this, and prompts for a message (optionally passed via the -m flag).

    Committing in a repo causes all staged files to return to being merely tracked and a subsequent git status will report no changes of the files staged.

    $ git status
[ ... ]
# Changes to be committed:
#       new file:   file1
[ ... ]
$ git commit -m "Added file1 to the repo"
[master (root-commit) e71f65b] Added file1 to the repo
 1 file changed, 0 insertions(+), 0 deletions(-)
 create mode 100644 file1

    Commits are used to document important changes in the repo, not just on the file level but also in so far as branching and merging are concerned.

    Commit guidelines

    • no whitespace at the start or end
    • should be logically distinct: every commit must do something separate (rebase if necessary before pushing)
    • Short (50 chars or less) summary of changes on first line followed by a blank line and longer explanation if neccessary.

    Modified and Unmodified files

    We have two last states you'll see if you look at a git status: modified and unmodified. A modified file has changes that aren't recorded in a past snapshot of the repository while a unmodified file has no changes that haven't been recored in a snapshot. To commit all modified files, simple plonk git commit -a in.

    Let's tie all of these states together:

    • When you start a repository, all files are untracked.
    • We then stage some files...
    • and then we commit. Now some files are tracked and unmodified while some are untracked.
    • We then work a bit on the project, modifying some of the files
    • We then decide to stage all the modified files along with some of the unmodified files.
    • And then commit them all

    History is not set in stone
    

    The scenario is easy to grasp and probably more common than we dare
  concede: You've just completed all your work, everything is working
  and now it's time to roll out the new revision. You git add your
  files and commit. Boom. Time to relax. A quick check of 'git log
  --summary' and you notice something. Oh no, you've left out
  'critical_file.c'. Oh dear.
    

    Perhaps you could just add another commit to say 'I
  forgot these files, um, sorry' but this isn't always desirable and
  it may annoy your repo-controlling overlords (see
  later). Furthermore, your mistake could have been as trivial as an
  incomplete commit message -- what then? To that
  end, git offers you recourse through 'git commit --amend'.
    

    Put simply, this command simply re-purposes the last
  commit to include all staged files and allows you to include
  potentially critical information in the commit message.

     1 file changed, 1 insertion(+)
     create mode 100644 even_moar

     create mode 100644 even_moar
     create mode 100644 extra

     commit 1c85d3b626707a9ded2250f99c183db0bfd2cf63

     Here is everything, for real this time

    Pulling your weight
    

    There are two important times when you would need your repo to
  catch-up to the server/master repo.
      
    
    • Important work upon which you depend has occured
      elsewhere and you wish to be up-to-date with it
      • 
    
    • You have done some work and wish to push your
    changes but cannot because other changes have occured elsewhere
      and are now on the server
    

     The first case is easy to handle. Merely run 'git
  pull' and all the changes will be downloaded to your repo and merged
  (though this is not guaranteed to be problem free)
    

    The second case begins in the same way, but comes with
  the high probability that you will need to manually merge files. In
  particular, when you pull because you wish to push your changes but
  cannot, all of the changes you do not have are downloaded into your
  local repo. In fact, there will be two 'branches' (think of them as
  concurrently existing versions of the content) sitting in your local
  repo.
    

    If you wish to unify your changes with the changes from
  elsewhere - and thus be able to push - you must merge these two
  branches (more on this later). In principle, you need to tell git to
  combine the work done elsewhere with your current work and commit
  that change to your repo so that there is only one version which
  represents the work done on both.
    
  
    
      
            $ git pull
        [ ... ]
        $ git commit -m "Merge of work from local/master and origin/master"
        $ git push
  
    

    


    

    While we're on the topic
    

    Branches are a fundamental component of a successful git
  workflow. They allow you to simultaneosly persue different avenues
  and maintain concurrent, incompatible or decidedly separate versions
  of your content (say, a testing and stable version) while being able
  to merge the one into the other when appropriate.
    

    It has been suggestively named, and so should be
  imagined as a literal branch in the timeline of changes that runs
  concurrently.
    

    The basic tools of the trade for branching
  are 
    • git branch [<new_branch_name>]
      • 
    
    • git checkout <branch>
      • 
    
    • git merge <branch>
    

    


    

    Branch
    

    The first port of call is that of 'git branch'
    
  
    
      
            $ git branch
        * master
  
    

    Without any arguments, it merely informs you of all the
  branches contained in the repo. However, if we provide it with an
  argument, it forms a branch in the timeline of changes under that
  name
    

    In actuality, virtually everything you would like to
  achieve in so far as manipulating branches is concenrned, is done
  through the branch command:
      
    
    • Rename the current branch: git branch -m <name>
      • 
    
    • Safe delete the current branch: git branch -d <branch>
      
        
        
      • Git considers it safe to remove a branch only
        when there are no outstanding merges. That is, only if its
          content exists elsewhere.
        • 
        
      • You'll see a polite error message informing
          you that there is content yet to be merged
        • 
    
      • 
    
    • Force delete (ye be warned): git branch -D <branch>
      • 
    
    • Gain insight into the structure and coherence of
    your branches:
        
        
      • You may wish to know precisely which branches
        are in-step with the branch you are currently working on
        (those that do not depend on changes which have not been made
        in your branch) or which branches are within reach of a
          particular commit. To this end: git branch --merged
        [<commit>]
        • 
        
      • The complement of this piece of information is
        oftentimes also useful (when merging, care needs to be taken
        with all branches which are not so aligned), thus there is:
          git branch --no-merged
    

    


    

    Checkout
    

    Using 'git checkout' we can effectively change universes and work
  within the safe environment of our branch. Here, any changes that we
  make and commit effect only this branch and not our original
  timeline.
    

    To that end, we are free to muck things up, perfect
  concepts or just generally persue other avenues or test features
  before they become part of the main body
    

    Git takes care of managing all the relevant files and
  directories for us, if something is part of master, and we checkout
  to experimental, it will no longer appear in a directory
  listing.
    

    We may maintain arbitrarily many distinct working
  environments in such a set-up
    

    


    

    Merge
    

    Possibly the most complex (not complicated) part of managing
  a git repo: combining branches.
    

    The basic command is 'git merge <branch>'. This
  will merge 'branch' into the current branch and automagically settle
  some rather complex under-the-hood things.

    There are two scenarios: a fast-forward merge and a
  three-way merge; git will do different things in each case. The only
  thing to take away from this is, if you want a commit to be
  generated every time you merge use the '--no-ff' flag when
  merging
    
 
    
      
            $ git merge experiments
        [ ... ]
        Auto-merging slides.html
        CONFLICT (content): Merge conflict in slides.html
        Automatic merge failed; fix conflicts and then commit the
        result.
        $ git status
        [ ... ]
        # Both modified: slides.html
 
    
 
    Sometimes this may happen to you, when you pull or
 when you merge. All git is saying is that the branches your are
 playing with have made potentially conflicting changes to the same
   file and it's not sure how to proceed.
    
 
    At this point, you should fire up your favourite
 editor and manually check the file in question. Git will neatly mark
 which bits came from what branch and it's up to you to choose the
 right ones.
 
    Once done, commit your changes and the branches
    have been successfully merged!
    

    


    

    Altogether now
    

      
  
    • Before we branch, we have some
  timeline representing the state of the repo
      • 
  
    • Branching introduces an
  alternate stream in which we may work, but does not move the head
    pointer to this state
      • 
  
    • By calling checkout we
  effectively switch to our parallel universe, git takes care of all
    the files which do or do not exist and the heead pointer is
  moved
      • 
  
    • We may then freely work in this
    branch without fear of upsetting our other work
      • 
  
    • If we wish to merge the features in
    the experimental branch, we switch to our main branch and call git
    merge
      • 
  
    • Now that we have merged our
    features, we may safely delete the experimental branch without git
    complaining. This is known as a fast forward merge as the master is an ancestor of the branch being merged which means we can simply move it up.
      • 

    

    
    
    

    

    


    

     Merging: a visual example 
    

      
  
    • Here we have a basic branch structure similar to what we have before.
      • 
  
    • We decide we want to add some experimental features, so we add a testing branch and switch to it.
      • 
  
    • We then work on it a bit
      • 
  
    • Suddenly, our supervisor demands we fix a bug in the master branch, so we check that out and do some work on it.  
      • 
  
    •  To try and placate our testy supervisor, we decide to merge some of the awesome testing features into our master branch, giving us a repo that looks like...
      • 
  
    • ...this, with the merge of the two branches becoming its own commit. This is called a three way merge and it is made based on the latest common ancestor the branches being merged share.
      • 

    

    
    
    

    

    


    

    Overall then
    

     We have seen how to use the basic building of git to create a repo
  and populate it, clone one and meddle in it, and push and pull
  changes
    

    On top of this structure, we imposed a higher level
  wherein all of this could easily happen inside a branch of project
  and happily remain disjoint until such a time that it was ready to
  be merged with the main stream
    

    All that remains is to chat about the most interesting
  concept, rebase, before a brief foray into workflow and actually using git.

    


    

    Rebase or why you would opaquely change the past
    

    Suppose the following situation occurs:
      
    
    • You branch your a remote repo and happily hack
      away at it
      • 
    
    • Upstream changes occur, and you decide to pull
    them in. 'No worries', you think, 'I have all my work in another
      branch. I can pull without conflict'. You do so.
      • 
    
    • There are some really interesting changes made in
    the master branch, and you think to yourself that it would be
      wonderful to have those in your branch.
      • 
    
    • How could you do this? Well, you could perform a
    (three-way) merge between your branch and the master one, generate
      a superfluous commit in the timeline and then proceed.
      • 

    

    Surely there is a better way to do this? Rebase is the
  'better' way. Instead of that last step, git rebase <base>
  (where base is anything that would pass-up as a reference to a
  particular point in the timeline of the project) takes your entire
  branch, subtly changes the files within it to be up-to-speed with
  the desired branch and then applies your commits to those files and
  finally pretends as though the whole thing never
  happened. Yes. That's right.
    

    There will be no trace whatsoever that you branched off
  at some earlier point then current records indicate and your branch
  will be ahead of the branch from which it came, generating a nice,
  linear flow.
    

    Rebasing is your way of saying 'Git, I want my branch
  to be based on the work that everyone else has already done to
  branch x'
    


    

     Rebasing: a visual example 
    

      
  
    • Here we have a similar situation to the
  previous merging example: we have two branches with changes that we want to
  put together. If we don't mind complexity, merging is a way to go. But if we
  want a nice linear order, we can do something a bit different
      • 
  
    • We're going to perform a rebase of Test onto Master:
      • 
  
    • Here we can see that instead of having a
  messy structure as before, we have one linear line. Note that c4' is a
  completely different commit than c4; it's c4 with all the changes from c3
  patched onto it. 
      • 
  
    • We can now just do a fast forward
  merge on master and we're done.
      • 

    

    
    
    

    

    


    

     Workflow 
    

     Since git is distributed, there's a multitude of different workflows. A
good outline of these is Pro Git by Scott Chacon
available free online at the link.
    


     We'll just cover briefly the model that is commonly used for small teams.
    


      
    
    •  You all commit to a central repository with the master branch being a
    stable version
      • 
    
    •  However, all work you do for a feature is on a separate branch. For
    example, if you were working on a game there'd be a branch for adding AI, a
    branch for working on game logic, another for collision detection and so
    on.
      • 
    
    •  When a feature is complete, you notify the branch manager that you
    want to merge. They then test it and, if satisfied, merge it into the main
    branch which then becomes the base for all other work.
      • 
    
    •  If when testing a bug appears in the master branch, you create a bug
    fix branch (e.g. hotfix1), fix the problem and then merge the changes back
    in.
      • 

    


    The advantages of this model is that you always have a clear division between
future features and currently functionality giving you the flexibility to try
new things without having the uncertainty and instability that having everyone
changing the main branch would create.
    


    Often this model is a bit more complex in that every developer uses a
different repository rather than branch, but the general pattern is the same.
